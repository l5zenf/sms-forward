下面给你一版**完整 MVP 实现计划**。这版不是单纯架构图，而是能直接拆任务、写代码、验收的计划。

目标：**树莓派 + Air780E + Rust/Kameo + SQLite/SeaORM，实现可靠短信接收、中文兼容、持久化队列、异步转发。**

---

# 1. MVP 总目标

做一个树莓派常驻服务：

```text
air780e-smsd
  ├─ 初始化 Air780E AT 串口
  ├─ 监听 +CMTI 新短信通知
  ├─ AT+CMGR 读取 PDU 短信
  ├─ 解码中文 / 长短信
  ├─ 先写入 SQLite
  ├─ 写入成功后 AT+CMGD 删除模块短信
  ├─ 标记 status=pending
  ├─ ForwarderActor 拉取 pending
  ├─ 转发成功标记 sent
  └─ 转发失败 pending 重试 / failed
```

核心原则：

```text
宁可重复转发，也不能丢短信。
```

# 2. 和 main.lua 初始化保持一致的点

原 `main.lua` 初始化里做了这些关键事：使用 `uart.VUART_0`，设置 `max_buffer_size=50`，调用 `mobile.setAuto(0)` 禁用自动数据连接，初始化 WDT，初始化 UART `115200 8N1`，注册短信、SIM、来电事件，注册 UART 命令解析，启动消息队列消费者，5 秒后发送 `system_ready`，每 60 秒发一次 `heartbeat`。([GitHub][1])

Rust MVP 要对齐这些语义：

| main.lua 初始化语义          | Rust/AT MVP 对应                    |
| ----------------------- | --------------------------------- |
| USB 虚拟串口通信              | 打开 Air780E AT 口 `/dev/ttyUSBx`    |
| 115200 8N1              | 串口配置 115200 8N1                   |
| 禁用自动数据连接                | 禁用 ModemManager，不启用 PPP/QMI/ECM   |
| 内存队列上限                  | Kameo bounded mailbox + SQLite 队列 |
| `SMS_INC` 事件            | `AT+CNMI=2,1,0,0,0` 监听 `+CMTI`    |
| `SIM_IND`               | `AT+CPIN?` 轮询/状态记录                |
| `CC_IND`                | MVP 预留 `AT+CLIP=1`，来电事件放 MVP-2    |
| WDT                     | systemd 重启 + HealthActor 心跳       |
| `system_ready`          | 初始化完成后记录 ready event              |
| 60 秒 heartbeat          | HealthActor 每 60 秒采集状态            |
| buffer overflow 4096 清理 | AT reader buffer 限制 4096/8192     |

## MVP-1 必须做

| 模块         | 必须实现                                                                |

| 长短信        | 支持 UDH 分片保存与合并                                                      |
| SQLite 持久化 | SeaORM migration + repository                                       |
| 状态机        | `pending/sending/sent/failed/decode_failed`                         |
| 可靠删除       | SQLite commit 成功后才 `AT+CMGD`                                        |
| Actor 模型   | Kameo：AtActor、SmsIngestActor、ForwarderActor、ReaperActor、HealthActor |
| 错误处理       | SNAFU 统一 AppError                                                   |
| 依赖组织       | `Arc<dyn Trait>` 注入 modem/repo/forwarder/decoder                    |
| 转发         | MVP 先只做 Webhook                                                     |

| 部署         | systemd 常驻、自动重启                                                     |

Kameo 的 Actor 使用 mailbox 接收消息，支持 bounded mailbox 做背压，适合这里把串口、入库、转发拆成独立异步 Actor。([Docs.rs][2]) SNAFU 的优势是给底层错误补业务上下文，比如同样是 IO 错误，可以区分“串口打开失败”“AT 超时”“数据库写入失败”。([Docs.rs][3]) SeaORM migration 的 `up/down` 可以管理建表、索引和回滚，适合这个 SQLite schema 管理。([SeaQL][4])

# 5. 阶段 0：实机 AT 验证

这是第一优先级。不要先写 Actor。

## 必须做

```bash
ls -l /dev/ttyUSB*
dmesg | grep -i tty
```

逐个测试：

```bash
sudo picocom -b 115200 /dev/ttyUSB0
sudo picocom -b 115200 /dev/ttyUSB1
sudo picocom -b 115200 /dev/ttyUSB2
```

找到能返回 `OK` 的 AT 口。

手动执行：

```text
AT
ATE0
AT+CMEE=2
AT+CPIN?
AT+CSQ
AT+CREG?
AT+CEREG?
AT+COPS?
```

短信初始化：

```text
AT+CMGF=0
AT+CPMS="SM","SM","SM"
AT+CNMI=2,1,0,0,0
```

`AT+CMGF` 用于设置短信输入/输出格式，`0` 是 PDU 模式，`1` 是 TEXT 模式；MVP 为了中文和长短信，固定用 PDU。([OpenLuat 文档][5]) `AT+CNMI` 用于设置新短信如何上报，`<mt>=1` 时新短信存储后通过 `+CMTI: <mem>,<index>` 通知 TE，官方示例就是 `AT+CNMI=2,1` 后收到 `+CMTI: "SM",1`。([OpenLuat 文档][6])

收到短信后测试：

```text
+CMTI: "SM",3

AT+CMGR=3

AT+CMGD=3
```

`AT+CMGD=<index>` 用于删除优选短信存储区里指定 index 的短信，所以后续程序必须读完并持久化后删除，避免 SIM 存储满。([OpenLuat 文档][7])

## 产出物

建立文档：

```text
docs/air780e-at-samples.md
```

记录：

```text
AT 口：/dev/ttyUSB?
SIM 状态：CPIN 返回
运营商：COPS 返回
信号：CSQ 返回
CNMI 配置返回
英文短信 PDU 样本
中文短信 PDU 样本
中文长短信多片 PDU 样本
```

## 验收标准

```text
[ ] 找到稳定 AT 口
[ ] AT 返回 OK
[ ] CPIN 返回 READY
[ ] CMGF=0 成功
[ ] CPMS="SM","SM","SM" 成功
[ ] CNMI=2,1,0,0,0 成功
[ ] 收到短信时出现 +CMTI
[ ] CMGR 能读出 PDU
[ ] CMGD 能删除短信
[ ] 保存中文短信 PDU 样本
[ ] 保存中文长短信 PDU 样本
```

## 关键疑难点

| 问题              | 指导                                          |
| --------------- | ------------------------------------------- |
| 不知道哪个是 AT 口     | 逐个 `/dev/ttyUSBx` 发 `AT`，不要猜                |
| ModemManager 抢口 | `sudo systemctl disable --now ModemManager` |
| 没有 `+CMTI`      | 检查 `AT+CNMI?`、SIM 是否 READY、短信存储是否满          |
| 中文乱码            | 不用 TEXT 模式，强制 PDU                           |
| 模块短信满           | 程序必须成功入库后 `CMGD`                            |

## 状态机

```text
pending
  ↓ claim
sending
  ├─ 转发成功 -> sent
  ├─ 转发失败且未超限 -> pending
  └─ 转发失败且超限 -> failed

decode_failed：PDU 已保存，但解码失败
```

## claim_next_pending 必须原子化

不要：

```text
SELECT pending
UPDATE sending
```

必须：

```text
BEGIN IMMEDIATE
SELECT 一条 pending
UPDATE 为 sending
COMMIT
```

SQLite 没有 PostgreSQL 的 `SKIP LOCKED`，MVP 里用 `BEGIN IMMEDIATE` 最直观。

## 验收标准

```text
[ ] migration 能创建所有表
[ ] 启动能创建 sms.db
[ ] save_single_message 后 status=pending
[ ] claim_next_pending 后 status=sending
[ ] mark_sent 后 status=sent
[ ] mark_failed 后 retry_count + 1
[ ] retry_count 超过 max_retry 后 status=failed
[ ] recover_stale_sending 能恢复 sending -> pending
[ ] dedupe_key 重复时不重复插入
[ ] sms_parts 到齐后能合并成 sms_messages
```

# 8. 阶段 3：AT Client + 初始化状态机

这是整个系统的关键。

## 设计原则

```text
只有 AtActor / AtClient 能拥有串口。
其他 Actor 不允许直接访问 serial。
HealthActor、SmsIngestActor、ForwarderActor 都不能直接发 AT。
```

原因：AT 响应和 URC 会混在同一个串口里。必须由一个统一 reader loop 解析。

## 初始化状态机

```text
OpenSerial
  ↓
ProbeAT
  ↓
DisableEcho
  ↓
EnableVerboseError
  ↓
WaitSimReady
  ↓
SetPduMode
  ↓
SetStorage
  ↓
SetNewMessageIndication
  ↓
SetClip
  ↓
InitialStatus
  ↓
Ready
```

## 初始化命令

```text
AT
ATE0
AT+CMEE=2
AT+CPIN?
AT+CMGF=0
AT+CPMS="SM","SM","SM"
AT+CNMI=2,1,0,0,0
AT+CLIP=1
AT+CSQ
AT+CREG?
AT+CEREG?
AT+COPS?
```

## Reader loop 分类

串口每读到一行，要分成三类：

```text
1. 当前 AT 命令响应
2. URC 异步事件
3. 噪声/空行/未知行
```

URC 至少支持：

```text
+CMTI: "SM",3
RING
+CLIP: ...
NO CARRIER
```

MVP-1 只处理 `+CMTI`，但 parser 结构要预留来电。

## buffer 上限

对齐 `main.lua` 的 4096 buffer 保护：

```text
read_buffer_limit = 4096 或 8192
超过上限：
  清空当前 buffer
  写 modem_events
  不 panic
```

## 验收标准

```text
[ ] 能打开配置里的 /dev/ttyUSBx
[ ] AT 返回 OK
[ ] ATE0 后无回显
[ ] CPIN 未 READY 时能等待重试
[ ] 初始化失败返回 SNAFU ModemInit 错误
[ ] +CMTI 能解析出 mem/index
[ ] AT 超时能返回 AtTimeout
[ ] buffer 超限能清空并记录事件
[ ] 初始化完成后产生 system_ready
```

## 关键疑难点与指导

| 难点              | 指导                                 |
| --------------- | ---------------------------------- |
| AT 响应和 URC 混在一起 | 必须统一 reader loop，不要每个函数自己 readline |
| 初始化期间来了短信       | URC 先缓存，Ready 后处理                  |
| 串口断开            | 关闭 fd，重新 open，重新执行 init            |
| 连续 AT 超时        | 先重开串口，失败再交给 systemd 重启             |
| SIM 不 READY     | 每 3 秒重试，最多等 60 秒                   |

# 9. 阶段 4：PDU 解码：中文 + 长短信

这是 MVP 最大难点。

## 必须解析字段

```text
SMSC length
SMSC
PDU type
sender address
PID
DCS
SCTS timestamp
UDL
UDH
user data
```

## 必须支持编码

| 编码              | MVP 要求       |
| --------------- | ------------ |
| UCS2 / UTF-16BE | 必须支持，中文短信核心  |
| GSM 7-bit       | 支持普通英文       |
| 8-bit           | 保存 raw，尽量不要丢 |
| UDH multipart   | 必须支持长短信      |
| WAP Push        | 不支持，raw 入库   |

## 中文短信

DCS 为 `0x08` 时，按 UCS2 / UTF-16BE 解码：

```text
4F60597D -> 你好
```

## 长短信 UDH

8-bit reference：

```text
05 00 03 XX TT SS
```

含义：

```text
XX = concat_ref
TT = total parts
SS = current sequence
```

16-bit reference：

```text
06 08 04 XX XX TT SS
```

## 处理流程

```text
PDU
  ↓
解析 sender / sms_time / dcs / udh / body
  ↓
如果无 UDH：
    save_single_message(status=pending)
  ↓
如果有 UDH：
    save_part()
    try_assemble_multipart()
    如果分片齐全：
        合并 content
        save_single_message(status=pending)
```

## 解码失败规则

```text
CMGR 成功拿到 raw_pdu
  ↓
raw_pdu 必须先入库
  ↓
解码失败 status=decode_failed
  ↓
再允许 CMGD 删除模块短信
```

否则坏短信会堵住 SIM 存储。

## 去重规则

不要用 modem index 去重，因为 index 会复用。

建议：

```text
part_dedupe_key =
sha256(iccid + sender + sms_time + concat_ref + concat_seq + pdu_raw)

message_dedupe_key =
sha256(iccid + sender + sms_time + full_content)
```

## 验收标准

```text
[ ] 英文普通短信能解码
[ ] 中文普通短信能解码
[ ] 中文长短信 2 片能合并
[ ] 中文长短信 3 片能合并
[ ] 重复收到同一分片不重复插入
[ ] 解码失败 raw_pdu 入库
[ ] 解码失败不会导致程序 panic
```

## 关键疑难点与指导

| 难点                 | 指导                                   |
| ------------------ | ------------------------------------ |
| GSM 7-bit + UDH 偏移 | 先做 UCS2，再做无 UDH 7-bit，最后做带 UDH 7-bit |
| 长短信乱序到达            | 按 `concat_seq` 排序合并                  |
| 分片缺失               | 只保存 part，不生成 pending message         |
| 重复分片               | `part_dedupe_key UNIQUE`             |
| 解码失败               | raw_pdu 落库，status=decode_failed      |

# 10. 阶段 5：Kameo Actor 主流程

## Actor 列表


## AtActor

职责：

```text
1. 初始化 Air780E
2. 独占串口
3. 监听 URC
4. 收到 +CMTI 后执行 AT+CMGR
5. 把 RawSmsPduReceived 发给 SmsIngestActor
6. 收到 SmsPersisted 后执行 AT+CMGD
7. 串口异常时重连
```

AtActor 内部需要维护：

```text
pending_delete:
  key = mem + index
  value = raw_pdu / received_at
```

原因：`SmsIngestActor` 是异步处理，处理完要通知 AtActor 删除对应 index。

## SmsIngestActor

职责：

```text
1. 接收 RawSmsPduReceived
2. 调 PduDecoder 解码
3. 保存 sms_parts 或 sms_messages
4. 解码失败保存 decode_failed
5. 成功或 raw 入库后回发 SmsPersisted 给 AtActor
```

## ForwarderActor

职责：

```text
1. 每 3 秒 claim_next_pending
2. status=pending -> sending
3. 调 WebhookForwarder
4. 成功 mark_sent
5. 失败 mark_failed
```

## ReaperActor

职责：

```text
每 60 秒执行 recover_stale_sending
```

把卡住的任务恢复：

```text
sending 且 locked_at 超过 5 分钟 -> pending
```

## HealthActor

职责：

```text
每 60 秒向 AtActor 发 QueryStatus
AtActor 串行执行：
  AT+CPIN?
  AT+CSQ
  AT+CREG?
  AT+CEREG?
  AT+COPS?
结果写 modem_status 和 modem_events
```

注意：HealthActor 不直接操作 modem。

## 消息定义

```rust
pub struct RawSmsPduReceived {
    pub mem: String,
    pub index: i32,
    pub raw_pdu: String,
}

pub struct SmsPersisted {
    pub mem: String,
    pub index: i32,
}

pub struct QueryStatus;

pub struct ForwardTick;

pub struct RecoverStaleSending;

pub struct HealthTick;
```

## 主链路

```text
AtActor
  ↓ +CMTI
AT+CMGR
  ↓
RawSmsPduReceived -> SmsIngestActor
  ↓
PDU decode + SQLite commit
  ↓
SmsPersisted -> AtActor
  ↓
AT+CMGD
```

## 转发链路

```text
ForwarderActor
  ↓
claim_next_pending
  ↓
WebhookForwarder.forward
  ↓
mark_sent / mark_failed
```

## 验收标准

```text
[ ] FakeModem 模拟 +CMTI
[ ] AtActor 调 read_sms_pdu
[ ] SmsIngestActor 能入库
[ ] 入库成功后 AtActor 调 delete_sms
[ ] ForwarderActor 能把 pending 转 sent
[ ] Webhook 失败时转 pending retry
[ ] ReaperActor 能恢复 stale sending
[ ] HealthActor 能周期写 modem_status
```

## 关键疑难点与指导

| 难点           | 指导                                                        |
| ------------ | --------------------------------------------------------- |
| Actor 互等导致死锁 | AtActor 发消息给 SmsIngestActor，SmsIngestActor 完成后回发，不要双向同步等待 |
| 多 Actor 抢串口  | AtActor 独占 modem                                          |
| mailbox 无限涨  | 使用 bounded mailbox                                        |
| DB 写失败       | 不发送 SmsPersisted，不删除模块短信                                  |
| PDU 解码失败     | raw 已入库才发送 SmsPersisted                                   |

# 11. 阶段 6：Webhook Forwarder + 状态机

## Webhook payload

```json
{
  "id": 1,
  "sender": "+447xxxx",
  "content": "验证码 123456",
  "sms_time": "2026-06-09T12:00:00Z",
  "received_at": "2026-06-09T12:00:02Z",
  "encoding": "ucs2",
  "dedupe_key": "..."
}
```

## 转发规则

```text
HTTP 2xx:
  mark_sent

HTTP 非 2xx:
  mark_failed

timeout:
  mark_failed

网络错误:
  mark_failed
```

## mark_failed 逻辑

```text
retry_count + 1 < max_retry:
  status = pending
  retry_count += 1
  next_retry_at = now + backoff

retry_count + 1 >= max_retry:
  status = failed
  retry_count += 1
```

backoff：

```text
delay = min(max_delay_secs, base_delay_secs * retry_count)
默认 base=60s，max=3600s，max_retry=10
```

## 幂等建议

Webhook payload 必须带：

```text
id
dedupe_key
```

因为可能出现：

```text
Webhook 实际成功
但本地 mark_sent 前进程崩溃
重启后再次转发
```

这个系统优先保证不丢，允许偶发重复。接收端最好按 `dedupe_key` 幂等。

## 验收标准

```text
[ ] 2xx -> sent
[ ] 500 -> pending retry
[ ] timeout -> pending retry
[ ] 超过 max_retry -> failed
[ ] kill -9 during sending -> 重启后 Reaper 恢复 pending
[ ] payload 包含 dedupe_key
```

```text

数据库写失败：

  记录 modem_event / app_event
```

# 15. 最终验收场景

## 场景 1：普通中文短信

```text
发送：验证码 123456
预期：
  +CMTI 被捕获
  CMGR 读取 PDU
  content = 验证码 123456
  sms_messages.status = pending
  CMGD 删除模块短信
  Webhook 成功
  status = sent
```

## 场景 2：中文长短信

```text
发送超过 70 个中文字符
预期：
  多个 PDU 分片进入 sms_parts
  分片齐全后合并
  sms_messages 生成一条完整短信
  status = pending -> sent
```

## 场景 3：Webhook 挂掉

```text
Webhook 返回 500 / timeout
预期：
  status = pending
  retry_count + 1
  next_retry_at 延后
  不丢短信
```

## 场景 4：进程转发中被杀

```text
status = sending 时 kill -9
重启后：
  ReaperActor 发现 stale sending
  恢复 pending
  ForwarderActor 继续转发
```

## 场景 5：PDU 解码失败

```text
CMGR 拿到 raw_pdu
decoder 返回错误
预期：
  raw_pdu 入库
  status = decode_failed
  CMGD 删除模块短信
  程序不崩
```

## 场景 6：数据库写失败

```text
SQLite 不可写 / 磁盘满
预期：
  不发送 SmsPersisted
  不执行 CMGD
  模块短信仍留在 SIM 存储
  日志有 Database error
```

# 16. 推荐开发顺序

不要按“架构优先”，要按风险最高的地方优先：

```text
1. 实机 AT 验证
2. PDU 样本收集
3. SQLite schema + repository
4. PDU decoder
5. AtClient 初始化 + +CMTI 监听
6. 单线程跑通：CMGR -> decode -> DB -> CMGD
7. 接 Kameo Actor
8. 接 Webhook Forwarder
9. 加 Reaper/Health/systemd
10. 做 kill -9 和断网测试
```

最重要的是第 1、4、5、6 步。Actor 不是难点，**PDU 中文长短信和 AT 串口状态机才是难点**。

# 17. 最容易翻车的点

| 风险               | 必须怎么处理                           |
| ---------------- | -------------------------------- |
| AT 口被系统抢         | 禁用 ModemManager                  |
| 多 Actor 同时写串口    | AtActor 独占串口                     |
| 先 CMGD 后入库       | 严禁，必须 DB commit 后 CMGD           |
| TEXT 模式中文乱码      | 强制 PDU                           |
| 长短信漏片            | sms_parts 单独存，齐全才生成 sms_messages |
| GSM 7-bit UDH 偏移 | 先保证 UCS2 中文；7-bit UDH 后置         |
| claim 非原子        | `BEGIN IMMEDIATE`                |
| Webhook 重复投递     | payload 带 dedupe_key             |
| SQLite 忙         | WAL + busy_timeout + 单写连接        |
| PDU 解码失败         | raw_pdu 入库，不 panic               |
